# 知识治理 — SSOT

> 本产品最关键也最危险的一环：用药/农事建议错了 = 真金白银 + 食品安全损失。
> 核心原则：**让 AI 做"组织和表达"，绝不让 AI 做"知识的源头"。**
> 版本：0.1.0 ｜ 当前处于「无人工审核员期」（见第 6 节）。

---

## 1. 三层知识来源模型（按可信度分层）

每条知识进库时必须标清属于哪一层、来自哪里。

### L1 · 权威固定知识（慢变、强约束、AI 不可编造）
| 类型 | 权威来源（中国/陕西桃树） |
|---|---|
| **农药用法（最关键）** | **中国农药信息网 ICAMA 登记数据库** — 哪些药登记可用于桃、靶标、稀释倍数、安全间隔期，一律以官方登记为准 |
| 技术规程 | 中国农科院**郑州果树研究所**（国内桃树最权威）、**西北农林科技大学**（杨凌，陕西果业最高学术权威） |
| 标准 | 国标 GB、农业行业标准 NY/T |
| 教材/专著 | 西农大果树栽培学教材、果树所技术手册 |

### L2 · 区域校准知识（结合当地）
- 陕西省农业农村厅 / 当地县农技推广站的本地技术意见、病虫情报
- 关中/陕北实际病虫害谱（与南方桃区不同）
- 当地实际物候历（比通用物候表更准）
- 12316 三农热线、当地农技公众号
- 当地用药习惯与抗药性历史

### L3 · 个体经验知识（有经验的人 + 自家）
- 当地老把式/土专家/合作社技术员经验
- 父母自己几十年经验
- 自家地块历史农事日志（去年这时打了什么、有没有效、产量多少）

### L0 · AI 草拟未审（临时层，见第 6 节）
- 由 GPT/Claude/MiniMax 起草、**尚未经人工校对**的内容。可用，但可信度最低、话术最保守、优先排在"待审核队列"。

## 2. "如何参考对的" — 四道校验关卡

一条知识进生产库要过四关：

1. **来源可信度** — 来自 L1/L2 权威？个人博客、电商详情、短视频"经验"一律降级或不收。
2. **时效性** — 农药登记/禁用名单每年都变。标采集日期，设复审周期；**用药类每年汛前重新核对登记库**。
3. **地域匹配** — 讲南方品种还是北方？气候海拔对得上关中/陕北吗？
4. **品种匹配** — 早/中/晚熟桃管理差别大，标清适用品种。

> 关键知识（尤其用药）需**交叉验证 2-3 个 L1 来源一致**才入库；不一致标"存疑"，走人工/农技员裁决，不直接给父母。

## 3. 溯源元数据 Schema（每条知识 chunk 必带）

进 chromadb / SQLite 时强制携带：

```json
{
  "text": "桃褐腐病：谢花后至采收期，雨后易发。防治可用戊唑醇...",
  "metadata": {
    "crop": "peach",
    "topic": "disease",            // disease/pest/fertilize/prune/irrigate/disaster/harvest
    "trust_level": "L1",           // L1/L2/L3/L0
    "source": "郑州果树研究所技术规程",
    "source_date": "2024",
    "region": ["关中", "陕北"],
    "variety": ["中熟桃"],
    "verified_by": "AI草拟未经人工审核",  // 有审核员后改为审核人姓名
    "review_status": "pending",    // pending / verified / disputed
    "review_due": "2026-03-01"
  }
}
```

检索逻辑硬规则：
1. **按 `trust_level` 加权**：L1 > L2 > L3 > L0。
2. **按 `region` / `variety` 过滤**：地域品种不匹配的直接排除。
3. 返回答案时**带出 `source`**，让父母/子女能看到"这条出处是哪"。

## 4. 用药安全硬规则（即使没有审核员也必须遵守）

⚠️ **农药推荐绝不走 LLM 生成。**

- 用药方案从一张**「已登记农药结构化表」**里查（`data/knowledge/pesticide_kb.jsonl`），这张表**人工从农药信息网录入**——因为它是"查表"不是"判断"，不能交给 AI。
- 查不到 → AI 回复"我这里没有登记可用于桃的对应药，建议咨询当地农技员"，**绝不瞎编药名/剂量**。
- 涉及**安全间隔期**的，强提醒 + 结合 D2 农事日志计算"距离上次打药/距离采收还有几天"。
- `src/ai/tasks.py` 的 `SYSTEM_DIAGNOSE` 已写入"不要编造未在参考知识中出现的农药名"，但这只是 prompt 层；**数据层（查表）才是真正的护栏**。

## 5. 经验沉淀机制（把"人脑里的"变成数据）

### 5.1 主动采访式录入（一次性，针对老把式/父母）
用问答清单采集（晚霜哪天来、一个长果枝留几个果、褐腐病雨后几天发用啥药），录音 → AI 转写结构化 → 有经验的人确认 → 入 L3。门槛降到"会说话就行"。

### 5.2 农事日志自动沉淀（持续，无感）
父母/子女每记一笔"打了 XX 药、疏了果"，就在积累这块地的专属知识库。一年后 AI 能说"你去年这时打过这个药，记录是有效"。

### 5.3 闭环反馈修正（让知识越用越准）
AI 给建议 → 父母执行 → 结果回流（好了/没好/更严重）→ 标注该条实际有效性 → 下次调整。**这是 L3 区别于死知识库的地方：会自我进化。**

## 6. 无人工审核员期的临时治理（当前阶段）⚠️

身边暂无可信农技审核员（需时间积累）。在此期间采取**保守模式**：

| 项 | 临时做法 |
|---|---|
| 知识来源 | 依赖 L1 权威文档 + AI 草拟，全部标 `review_status: pending`、`trust_level: L0/L1` |
| 用药表 | **唯独这张表必须人工查录**农药信息网，不接受 AI 草拟（哪怕没人复核，也比 AI 编造安全） |
| 话术 | 全面保守：拿不准就导向"问农技员/补拍"，App 常驻免责声明 |
| 替代审核 | 用**反馈闭环（5.3）** + **子女人工判断**临时替代人工审核关 |
| 待审队列 | 所有 `pending` 知识进 backlog；将来有审核员，对着 backlog 批量校验转 `verified` |

**免责声明（App 内常驻）**：
> 本应用建议由 AI 结合公开农技资料生成，**仅供参考**。用药请以农药标签说明和当地农技部门指导为准。涉及重大决策请咨询专业农技员。

## 7. 知识治理流程（含人在回路）

```
采集（L1官方 / L2区域 / L3经验）
  → AI 结构化 + 自动打溯源标签
  → 【人工审核关】※当前缺审核员，暂用反馈闭环+子女判断替代，标 pending
  → 入生产库
  → AI 检索时受 trust_level / region 约束生成
  → 父母使用 → 反馈回流（5.3）
  → 定期复审（用药每年汛前必复审）
```

## 8. MVP 知识范围（窄而准 >> 宽而错）

| 阶段 | 范围 | 做法 |
|---|---|---|
| **P0** | ① 关中中熟桃物候历（种子已建，父母校一遍即成 L3 准数据）② 本地最常发 5-8 种病虫图文 ③ 这几种对应的**已登记农药表**（人工录几十条） | AI 起草 → 你+父母校对 → 入库带溯源 |
| **P1** | 施肥/灌溉/修剪/花果管理规程（L1 西农大+果树所） | 人工校对入库 |
| **P2** | 农事日志沉淀 + 反馈闭环上线，知识开始自进化 | 工程闭环 |

> **第一版用药表宁可只覆盖本地最常见那几种病虫、几十条精校数据，也不让 AI 自由发挥几百条没人审的。**
