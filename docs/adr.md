## ADR-001 部署形态 ✅
- **背景**：个人/家庭项目，自掏腰包、自维护；父母在陕西用手机访问。
- **决定**：MVP 用 **家里 PC 跑 FastAPI + 内网穿透（cpolar/zerotier 等）** 暴露给父母手机；验证有效后再上**轻量云服务器**。
- **理由**：零额外成本快速验证；PC 关机即停的缺点在验证期可接受。
- **影响**：上线初期需保证家里 PC 常开；长期迁云要把数据/配置做成可整体搬迁。

## ADR-002 数据存储 ✅
- **背景**：要存地块、农事日志、用户绑定、知识元数据等结构化数据。
- **决定**：用 **SQLite**（单文件、零运维）。知识向量用 **chromadb**（本地持久化）。
- **理由**：比 JSON 文件强（事务/查询/并发），比独立数据库轻（无需部署）。够 MVP~中期用量。
- **影响**：所有持久化走 SQLite；迁云时单文件可直接带走。避免一上来上 Postgres 的运维负担。

## ADR-003 账户体系 ✅
- **背景**：需要父母 + 子女双角色，但老人产品不能有复杂注册。
- **决定**：用 **微信 openid 绑定** 作为身份，最轻量；不做独立用户名密码。
- **理由**：父母已有微信；扫码即绑定，零学习。
- **影响**：身份模型围绕 openid；`identity` 领域维护 `farmer`/`caretaker`/`农技员` 角色与绑定关系。

## ADR-004 多作物数据模型 ✅
- **背景**：MVP 做桃树，最终要扩苹果/梨/樱桃。
- **决定**：**所有领域数据 day-1 即带 `crop` 字段**，桃树只是第一个值。
- **理由**：零成本现在做；硬编码 peach 后扩展要大改（高成本以后补）。
- **影响**：知识、物候、日历、诊断、地块全部按 `crop` 隔离与检索。

## ADR-005 前端技术栈 ✅
- **背景**：移动端 H5，用户是老人。
- **决定**：**Vue3 + Vite + Vant + PWA**。
- **理由**：Vant 移动组件成熟、易做大字大按钮；PWA 支持"加桌面"+离线壳。
- **影响**：前端独立工程；需配套**适老化设计规范**（字号下限、按钮尺寸、对比度、语音覆盖率）作为硬约束。

## ADR-006 语音方案 ✅
- **背景**：父母不爱打字，倾向语音；但要控成本和复杂度。
- **决定**：ASR（语音转文字）先用**浏览器自带 Web Speech API**；TTS（朗读）用 MiniMax/MiMo provider 并**缓存音频**。
- **理由**：省掉一个云 ASR provider；TTS 缓存避免重复合成成本（可借姊妹项目 `tts_cache.py`）。
- **影响**：现有 provider 无需实现 ASR；若浏览器 ASR 效果不足，再评估云 ASR。

## ADR-007 方言处理 ✅
- **背景**：陕西父母说关中方言，通用 ASR 识别率会打折。
- **决定**：MVP **接受识别率损失**，用通用 ASR，并以**按钮/选择题**降低对语音输入的依赖。
- **理由**：方言模型投入大、周期长，不值得在 MVP 投入；按钮兜底能保证可用性。
- **影响**：交互设计上语音是"加分项"非"唯一项"，关键操作都要有可点选的替代路径。

## ADR-008 AI Provider 复用 ✅（vision 已验证）
- **背景**：姊妹项目「资料浏览器」已有成熟 provider 抽象。
- **决定**：**复制** `src/ai/`（base/factory/minimax/mimo/minimax_anthropic）到本项目，不改接口；仅在 `tasks.py` 加农业领域任务。
- **理由**：Capability 已分 TEXT/VISION/TTS/ASR，正好覆盖需求；避免重复造轮子，又不与姊妹项目代码绑死。
- **影响 / 验证结论（2026-06，用真实 MiniMax key 实测）**：
  - **文字**：用原生 `minimax` 的 `MiniMax-Text-01`，即出正文、快。
  - **vision 已确认可用**：正确模型名是 **`abab7-chat-preview`**；旧名 `MiniMax-VL-01` 已失效（返回 2013 unknown model）。`abab6.5s-chat` 实测看不到图。
  - **思考模型坑**：`minimax_anthropic`（M2.7）是 thinking 模型，max_tokens 小时 token 全耗在 thinking、正文为空 → 故 active 不用它。
  - **TTS / MiMo（已打通）**：MiMo 的 **Token Plan**（key 以 `tp-` 开头）必须用**专属 base_url** `https://token-plan-cn.xiaomimimo.com/v1`，与按量计费地址 `https://api.xiaomimimo.com/v1` 是两套独立的 key+地址，配错就 401。一个 Token Plan 订阅**同时覆盖文字(LLM)与 TTS**，共享额度。本项目据此给 `mimo.py` 增加了 `chat()`（TEXT 能力）并支持可配置 base_url。实测：文字 `mimo-v2.5`、TTS `mimo-v2.5-tts` 均可用。
  - **MiMo 思考模型坑**：`mimo-v2.5`/`mimo-v2.5-pro` 是 reasoning 模型，答案在 `content`、思维在 `reasoning_content`；max_tokens 太小会导致 content 为空，需给足 token。故 active 文字仍用 MiniMax-Text-01（无思考、更快）。
  - **能力归属结论**：文字=minimax(MiniMax-Text-01)；vision=minimax(abab7-chat-preview)；TTS=mimo(token-plan)。
  - ASR 接口仍无实现（见 ADR-006）。

## ADR-009 灾害预警逻辑 ✅
- **背景**：霜冻/冰雹漏报 = 绝收，后果极重。
- **决定**：灾害判定用**硬规则**（气温阈值/预报关键词）兜底，**不依赖 LLM 判断**；LLM 只负责把预警"讲清楚 + 给应对方案"。
- **理由**：极端事件不能赌 LLM 的可靠性。
- **影响**：`alert_scheduler` 写确定性规则；AI 仅做表达层（`explain_alert`）。

## ADR-010 用药建议来源 ✅
- **背景**：用药错误风险最高，且当前无人工审核员。
- **决定**：用药方案**只从人工查录的「已登记农药表」检索**，AI 不生成药名/剂量；查不到导向农技员。详见 `knowledge-governance.md` 第 4 节。
- **理由**：法律 + 食品安全底线，是整个产品最硬的护栏。
- **影响**：`pesticide_kb.jsonl` 必须人工维护；诊断/问答涉及用药时走查表而非生成。

## ADR-011 多作物架构 ✅
- **背景**：MVP 先做桃，但最终要苹果/梨/樱桃；早期代码硬编码了 `peach`，是扩展债。
- **决定**：知识按作物分目录 `data/knowledge/<crop>/{phenology,calendar,pests,pesticide,playbooks}`，
  加 `crops.json` 注册表。代码侧用 `CropRegistry`（domain）+ `KnowledgeManager`（按作物缓存 KB），
  所有路由用 `?crop=` 解析（非法值回退默认作物）。各作物**共用同一套物候期 key**，只是日期/内容不同。
- **理由**：扩展新作物 = 丢一个数据文件夹 + 注册一行，**零业务代码改动**；业务逻辑（日历/预警/诊断/问答）天然按作物分流。
- **影响**：`KnowledgeBase` 改为按作物隔离（去掉内部 crop 过滤）；诊断 prompt、mock 均按作物动态生成；
  已用 apple 验证（同一天 桃=膨大期、苹果=幼果期；霜冻预警严重度因物候不同而异）。
